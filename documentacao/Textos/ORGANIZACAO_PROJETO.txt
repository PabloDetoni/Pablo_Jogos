ORGANIZAÇÃO DO PROJETO — REGRAS PRINCIPAIS

IMPORTANTE: ESTE PROJETO DEVE CONTER APENAS TRÊS ÁREAS PRINCIPAIS: BACKEND, FRONTEND E DOCUMENTACAO. TUDO QUE NÃO PERTENCER A UMA DESTAS TRÊS PASTAS DEVE SER MOVIDO, ARQUIVADO OU REMOVIDO.

1) VISÃO GERAL RÁPIDA
- Objetivo: projeto didático de jogos online (Node.js/Express + PostgreSQL no BACKEND, frontend estático no FRONTEND).
- Regra de ouro: apenas as pastas BACKEND, FRONTEND e DOCUMENTACAO existem no nível superior do repositório.

2) ESTRUTURA OBRIGATÓRIA (NÍVEL SUPERIOR)
- /BACKEND  -> tudo relacionado ao servidor, APIs, banco de dados, scripts de inicialização e utilitários do servidor.
- /FRONTEND -> toda a aplicação estática (html, css, js, assets: img, CSVs, etc.).
- /DOCUMENTACAO -> explicações, manuais, scripts de migração e arquivos de texto explicativos. Tudo em texto legível (MD ou TXT).

OBS: Nomes das pastas devem estar em minúsculas no sistema (aqui usamos maiúsculas apenas para ênfase no texto), o diretório físico atual está em `backend/`, `frontend/`, `documentacao/`.

3) O QUE COLOCO EM CADA PASTA (DETALHAMENTO)

BACKEND (conteúdo mínimo esperado)
- server2.js (entrypoint do servidor) — deve ser o script start no package.json.
- database.js (pool e helpers query/transaction/testConnection).
- routes/ (arquivo por recurso, ex: jogoRoutes.js, usuarioRoutes.js, partidaRoutes.js, adminRoutes.js).
- controllers/ (lógica das rotas; controllers finos que chamam models e retornam status apropriados).
- models/ (opcional, helpers DB reutilizáveis: getAll, create, getById).
- utils/ (scripts helpers, ex: jogoFileManager.js, exportável e usado por controllers).
- setup/ (apenas arquivos de suporte: README, scripts de teste/local setup — já criado como `backend/setup`).
- package.json, package-lock.json, .env (variáveis de ambiente), node_modules/ (ou .gitignore para ignorar) — mantidos na raiz do BACKEND.

FRONTEND (conteúdo mínimo esperado)
- html/ (pasta com `estrutura/` para templates e `jogos/` para páginas geradas de cada jogo)
- css/ (idem: `estrutura/` e `jogos/`)
- js/ (idem: `estrutura/` e `jogos/`)
- img/, CSV/ ou assets/ — recursos usados pelos jogos.

DOCUMENTACAO (conteúdo mínimo esperado)
- Textos/ (resumos, instruções, checklist, arquitetura). Ex.: `ORGANIZACAO_PROJETO.txt`, `RJHALMEMAN_BACKEND_SUMMARY.txt`.
- SCHEMA_PADRAO.sql e demais scripts para inicializar o banco.

- Rodar `npm start` dentro de `backend/` (após atualizarmos o start para server2.js).
- Verificar `http://localhost:3001/health` para checar conexão com o banco.
